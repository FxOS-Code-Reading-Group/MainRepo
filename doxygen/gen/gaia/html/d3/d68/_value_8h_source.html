00821 /*
00822  * JS::Value is the interface for a single JavaScript Engine value.  A few
00823  * general notes on JS::Value:
00824  *
00825  * - JS::Value has setX() and isX() members for X in
00826  *
00827  *     { Int32, Double, String, Boolean, Undefined, Null, Object, Magic }
00828  *
00829  *   JS::Value also contains toX() for each of the non-singleton types.
00830  *
00831  * - Magic is a singleton type whose payload contains a JSWhyMagic "reason" for
00832  *   the magic value. By providing JSWhyMagic values when creating and checking
00833  *   for magic values, it is possible to assert, at runtime, that only magic
00834  *   values with the expected reason flow through a particular value. For
00835  *   example, if cx->exception has a magic value, the reason must be
00836  *   JS_GENERATOR_CLOSING.
00837  *
00838  * - The JS::Value operations are preferred.  The JSVAL_* operations remain for
00839  *   compatibility; they may be removed at some point.  These operations mostly
00840  *   provide similar functionality.  But there are a few key differences.  One
00841  *   is that JS::Value gives null a separate type. Thus
00842  *
00843  *           JSVAL_IS_OBJECT(v) === v.isObjectOrNull()
00844  *       !JSVAL_IS_PRIMITIVE(v) === v.isObject()
00845  *
00846  *   Also, to help prevent mistakenly boxing a nullable JSObject* as an object,
00847  *   Value::setObject takes a JSObject&. (Conversely, Value::asObject returns a
00848  *   JSObject&.)  A convenience member Value::setObjectOrNull is provided.
00849  *
00850  * - JSVAL_VOID is the same as the singleton value of the Undefined type.
00851  *
00852  * - Note that JS::Value is 8 bytes on 32 and 64-bit architectures. Thus, on
00853  *   32-bit user code should avoid copying jsval/JS::Value as much as possible,
00854  *   preferring to pass by const Value &.
