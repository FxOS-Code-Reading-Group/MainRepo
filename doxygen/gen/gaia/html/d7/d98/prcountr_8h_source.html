00009 /*----------------------------------------------------------------------------
00010 ** prcountr.h -- NSPR Instrumentation counters
00011 **
00012 ** The NSPR Counter Feature provides a means to "count
00013 ** something." Counters can be dynamically defined, incremented,
00014 ** decremented, set, and deleted under application program
00015 ** control.
00016 **                                                                                 
00017 ** The Counter Feature is intended to be used as instrumentation,                  
00018 ** not as operational data. If you need a counter for operational                  
00019 ** data, use native integral types.                                                
00020 **                                                                                 
00021 ** Counters are 32bit unsigned intergers. On overflow, a counter                   
00022 ** will wrap. No exception is recognized or reported.                              
00023 **                                                                                 
00024 ** A counter can be dynamically created using a two level naming
00025 ** convention. A "handle" is returned when the counter is
00026 ** created. The counter can subsequently be addressed by its
00027 ** handle. An API is provided to get an existing counter's handle
00028 ** given the names with  which it was originally created. 
00029 ** Similarly, a counter's name can be retrieved given its handle.
00030 ** 
00031 ** The counter naming convention is a two-level hierarchy. The
00032 ** QName is the higher level of the hierarchy; RName is the
00033 ** lower level. RNames can be thought of as existing within a
00034 ** QName. The same RName can exist within multiple QNames. QNames
00035 ** are unique. The NSPR Counter is not a near-zero overhead
00036 ** feature. Application designers should be aware of 
00037 ** serialization issues when using the Counter API. Creating a
00038 ** counter locks a large asset, potentially causing a stall. This
00039 ** suggest that applications should create counters at component
00040 ** initialization, for example, and not create and destroy them
00041 ** willy-nilly. ... You have been warned.
00042 ** 
00043 ** Incrementing and Adding to counters uses atomic operations.
00044 ** The performance of these operations will vary from platform
00045 ** to platform. On platforms where atomic operations are not
00046 ** supported the overhead may be substantial.
00047 ** 
00048 ** When traversing the counter database with FindNext functions,
00049 ** the instantaneous values of any given counter is that at the
00050 ** moment of extraction. The state of the entire counter database
00051 ** may not be viewed as atomic.
00052 ** 
00053 ** The counter interface may be disabled (No-Op'd) at compile
00054 ** time. When DEBUG is defined at compile time, the Counter
00055 ** Feature is compiled into NSPR and applications invoking it.
00056 ** When DEBUG is not defined, the counter macros compile to
00057 ** nothing. To force the Counter Feature to be compiled into an
00058 ** optimized build, define FORCE_NSPR_COUNTERS at compile time
00059 ** for both NSPR and the application intending to use it.
00060 ** 
00061 ** Application designers should use the macro form of the Counter
00062 ** Feature methods to minimize performance impact in optimized
00063 ** builds. The macros normally compile to nothing on optimized
00064 ** builds.
00065 ** 
00066 ** Application designers should be aware of the effects of
00067 ** debug and optimized build differences when using result of the
00068 ** Counter Feature macros in expressions.
00069 ** 
00070 ** The Counter Feature is thread-safe and SMP safe.
00071 ** 
00072 ** /lth. 09-Jun-1998.
00073 */
