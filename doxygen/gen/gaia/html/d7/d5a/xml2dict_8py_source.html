00015 The sample.xml file from the Android repo looks like this:
00016 
00017 ----------------------------------------------------------------------
00018 
00019     <!-- This is a sample wordlist that can be converted to a binary
00020          dictionary for use by the Latin IME. The format of the word
00021          list is a flat list of word entries. Each entry has a frequency
00022          between 255 and 0. Highest frequency words get more weight in
00023          the prediction algorithm. As a special case, a weight of 0 is
00024          taken to mean profanity - words that should not be considered a
00025          typo, but that should never be suggested explicitly. You can
00026          capitalize words that must always be capitalized, such as
00027          "January". You can have a capitalized and a non-capitalized
00028          word as separate entries, such as "robin" and "Robin". -->
00029 
00030     <wordlist>
00031       <w f="255">this</w>
00032       <w f="255">is</w>
00033       <w f="128">sample</w>
00034       <w f="1">wordlist</w>
00035     </wordlist>
00036 ----------------------------------------------------------------------
00037 
00038 This script processes the word list and converts it to a Ternary
00039 Search Tree (TST), as described in
00040 gaia/apps/keyboard/js/imes/latin/predictions.js and also in
00041 
00042   http://en.wikipedia.org/wiki/Ternary_search_tree
00043   http://www.strchr.com/ternary_dags
00044   http://www.strchr.com/dawg_predictive
00045 
00046 Note that the script does not convert the tree into a DAG (by sharing
00047 common word suffixes) because it cannot maintain separate frequency
00048 data for each word if the words share nodes.
00049 
00050 This script balances the TST such that at any node the
00051 highest-frequency word is found by following the center pointer. The
00052 script also overlays a linked list on top of the tree. At any node,
00053 the next most frequent word with the same parent node is found by
00054 following the next pointer.
00055 
00056 After building the TST data strucure this script serializes it into a
00057 compact binary file with variable length nodes. The file begins with
00058 the 8 ASCII characters "FxOSDICT" and four more bytes. Bytes 8, 9 and
00059 10 are currently unused and byte 11 is a dictionary format version
00060 number, currently 1.
00061 
00062 Byte 12 file specifies the length of the longest word in the
00063 dictionary.
00064 
00065 After these first 13 bytes, the file contains a character table that
00066 lists the characters used in the dictionary. This table is a two-byte
00067 big-endian integer that specifies the number of entries in the
00068 table. Each table entry is a big-endian two-byte character code
00069 followed by a big-endian 4-byte number that specifies the number of
00070 times the character appears in the dictionary.
00071 
00072 After the character table (starting at byte 15 + num_entries*6), the
00073 file consists of serialized nodes. Each node is betwen 1 byte and 6
00074 bytes long, encoded as follows.
00075 
00076 The first byte of each node is always an 8-bit bitfield: csnfffff
00077 
00078 The c bit specifies whether this node represents a character. If c is
00079 1 then a character code follows this first byte. If c is 0 then this
00080 is a terminal node that marks the end of the word and it consists
00081 of this single byte by itself.
00082 
00083 The s bit specifies the size of the character associated with this
00084 node. If s is 0 the character is one byte long. If s is 1 the
00085 character is a big-endian two byte value.
00086 
00087 The n bit specifies whether this node includes a next pointer. If n is 1 then
00088 the character code is followed by a big-endian 3 byte number.
00089 
00090 The fffff bits are represent a number between 0 and 31 and provide a
00091 weight for the node. This is usually based on the word frequency data
00092 from the dictionary, though this may be tuned by adjusting frequency
00093 depending on word length, for example. At any node, these frequency
00094 bits represent the weight of the highest frequency word under the
00095 node. (And, as described in the predictions.js file, the tree is
00096 balanced so that that highest frequency word is found by following the
00097 chain of center pointers.)
00098 
00099 If the c bit is set, the next one or two bytes (depending on the
00100 s bit) of the node are the Unicode character code that is stored in
00101 the node. Two-byte codes are big-endian.
00102 
00103 If the n bit was set, the next 3 bytes are a big-endian 24-bit
00104 unsigned integer offset to the start of the node pointed to by the
00105 next pointer.
00106 
00107 If the c bit was set the node has a character, and this means that it
00108 also has a center pointer. We serialize the tree so that a node's
00109 center pointer always points to the next node sequentially. So we
00110 never need to write the center pointer to the file: it is always the
00111 next node.
