00002 /*
00003  * This module is in charge of keep the historical of SIM changes in order
00004  * to rebuild the accurate usage of each SIM.
00005  *
00006  * This is needed due to two Statistics API limitations:
00007  * 1- There is no distinction per SIM in BE
00008  * 2- The minimum resolution is 24 hours.
00009  *
00010  * Bascically, the data structure is a list of tags. A tag is an object with a
00011  * SIM id, start and end dates that labels the usage during this period to
00012  * belong the SIM with that id.
00013  *
00014  * Take a look to this example:
00015  * ============================
00016  *
00017  * [{sim:"A", start: 1, actual: 2000,               offset: 2000, end:  2},
00018  *  {sim:"B", start: 2, actual: 5000, fixing: 3000, offset: 2000, end:  3},
00019  *  {sim:"A", start: 3, actual: 6000,               offset: 3000, end:  4},
00020  *  {sim:"B", start: 4, actual:16000 ,              offset:12000, end: 72},
00021  *  {sim:"A", start:72}]
00022  *
00023  * Each row is a Tag. Suppose times are hours in a day. Let's verbose these
00024  * lines:
00025  *
00026  * First row means from time 1 to 2, usage belongs to A and when A was removed
00027  * there was a real ussage of 2000 bytes and A consumed 2000 bytes as well.
00028  *
00029  * Second row means from time 2 to 3, usage belongs to B. B was reset when a
00030  * real consumption of 3000 bytes was done and removed when actual consumption
00031  * of 5000 was. So the total for B is 5000 - 3000 = 2000, the offset value.
00032  *
00033  * Third means from time 3 to 4, usage belongs to A again and, at the moment
00034  * of switching, the real usage was 6000 so the SIM usage is the new usage
00035  * from the former SIM to this (6000 - 5000 = 1000) plus the usage the same
00036  * SIM had before, this is, the last offset of the last tag with same SIM
00037  * (1000 + 2000 = 3000).
00038  *
00039  * Fourth means from time 4 to 72, usage belongs to B again.
00040  * Current value: 16000
00041  * Offset value: new usage + last offset = (16000 - 6000) + 2000 = 12000
00042  *
00043  * The last one means sim A was inserted at time 48 and is currently inserted.
00044  *
00045  * This example show how the limitations affect the frontend. Limitation 1 avoid
00046  * us to make a API reset because we were erasing all data including that not
00047  * belonging to us.
00048  *
00049  * Limitation 2 compels us to get the total usage by day. There is no doubt
00050  * about which SIM used the mobile interface for day 2 (time 24 to 48). It was
00051  * SIM B but, what happend during day 1 (time 0 to 24) or day 3 (time 48 to 72)?
00052  *
00053  * The module involves three mechanisms:
00054  * =====================================
00055  *
00056  * 1- Building the list of tags
00057  * ----------------------------
00058  * In order to build the list of tags, the module should be informed as soon as
00059  * the device is started to check if a SIM change was made. If so, the last tag
00060  * is closed and a new one is open.
00061  *
00062  * Closing a tag is to compute current and offset values and set the end date.
00063  * Opening a tag is to add a new tag object with the SIM id and the start date.
00064  *
00065  * When closing, we get tags labelling the current date. We say a tag labels a
00066  * date if:
00067  *   a) tag's start or end refer to the same day as date or
00068  *   b) date is between tag's start and tag's end or
00069  *   c) date is latter than tag's start and there is no tag's end
00070  *
00071  * At least one (the last and current) is retrieved. Now:
00072  *
00073  * > current value is got by asking Statistics API for the actual value at the
00074  *   moment of the switch.
00075  *
00076  * > offset value is calculated by:
00077  *   a) if there is a fixing value for now -> offset = actual - fixing
00078  *   b) if there is only one tag           -> offset = actual
00079  *   c) if there is more than one,         -> offset =
00080  *      (actual - <previous tag's current>) + <last tag with same SIM's offset>
00081  *
00082  * 2- Setting a fixing value
00083  * -------------------------
00084  * To simulate a reset, we need to take in count the amount spent at that moment
00085  * and keep it to correct the consumption of that SIM. This is done by setting
00086  * the fixing value to the real data spent at the moment of the reset.
00087  * This operation is not here but in the common.js module due to should be
00088  * accessed from several entry points of the application. Take a look at
00089  * resetData() method.
00090  *
00091  * In addition, when a reset is done, former references to the same SIM in the
00092  * tag list are deleted keeping the list short.
00093  *
00094  * 3- Getting the usage for a SIM in a goal date
00095  * ---------------------------------------------
00096  * To get this value we need to retrieve an ordered subset of the tag list
00097  * labelling the goal date.
00098  *
00099  * Now passing through this subset in reverse order we look for the first tag
00100  * with the SIM required. Several cases can occur:
00101  *   a) the tag has no end or the end is not the goal date, -> usage =
00102  *
00103  *     (actual - <previous tag's actual>*) + <last tag with same SIM's offset>**
00104  *       if there is no fixing value for the goal date
00105  *
00106  *     actual - <fixing value for goal date>
00107  *       if there is a fixing value for the goal date
00108  *
00109  *   ( *) Or 0 if there is no previous tag
00110  *   (**) Or 0 if there is no a closed tag
00111  *
00112  *   b) the tag is closed -> usage = tag's offset
00113  *
00114  * Conclussions
00115  * ============
00116  * With these 3 mechanisms we can provide accurate data usage statistics about
00117  * which SIM spent what data in the device. It is impossible to know about
00118  * consumption out of the device.
00119  *
00120  * Current implementation is not optimal in terms of space not time but it
00121  * has been made to meet scheduling constrins.
00122  */
