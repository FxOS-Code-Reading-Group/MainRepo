00141 /* For SSL 3.0 and TLS 1.0, by default we prevent chosen plaintext attacks
00142  * on SSL CBC mode cipher suites (see RFC 4346 Section F.3) by splitting
00143  * non-empty application_data records into two records; the first record has
00144  * only the first byte of plaintext, and the second has the rest.
00145  *
00146  * This only prevents the attack in the sending direction; the connection may
00147  * still be vulnerable to such attacks if the peer does not implement a similar
00148  * countermeasure.
00149  *
00150  * This protection mechanism is on by default; the default can be overridden by
00151  * setting NSS_SSL_CBC_RANDOM_IV=0 in the environment prior to execution,
00152  * and/or by the application setting the option SSL_CBC_RANDOM_IV to PR_FALSE.
00153  *
00154  * The per-record IV in TLS 1.1 and later adds one block of overhead per
00155  * record, whereas this hack will add at least two blocks of overhead per
00156  * record, so TLS 1.1+ will always be more efficient.
00157  *
00158  * Other implementations (e.g. some versions of OpenSSL, in some
00159  * configurations) prevent the same attack by prepending an empty
00160  * application_data record to every application_data record they send; we do
00161  * not do that because some implementations cannot handle empty
00162  * application_data records. Also, we only split application_data records and
00163  * not other types of records, because some implementations will not accept
00164  * fragmented records of some other types (e.g. some versions of NSS do not
00165  * accept fragmented alerts).
00166  */
00167 #define SSL_CBC_RANDOM_IV 23
00168 #define SSL_ENABLE_OCSP_STAPLING       24 /* Request OCSP stapling (client) */

00260 /* SSL Version Range API
00261 **
00262 ** This API should be used to control SSL 3.0 & TLS support instead of the
00263 ** older SSL_Option* API; however, the SSL_Option* API MUST still be used to
00264 ** control SSL 2.0 support. In this version of libssl, SSL 3.0 and TLS 1.0 are
00265 ** enabled by default. Future versions of libssl may change which versions of
00266 ** the protocol are enabled by default.
00267 **
00268 ** The SSLProtocolVariant enum indicates whether the protocol is of type
00269 ** stream or datagram. This must be provided to the functions that do not
00270 ** take an fd. Functions which take an fd will get the variant from the fd,
00271 ** which is typed.
00272 **
00273 ** Using the new version range API in conjunction with the older
00274 ** SSL_OptionSet-based API for controlling the enabled protocol versions may
00275 ** cause unexpected results. Going forward, we guarantee only the following:
00276 **
00277 ** SSL_OptionGet(SSL_ENABLE_TLS) will return PR_TRUE if *ANY* versions of TLS
00278 ** are enabled.
00279 **
00280 ** SSL_OptionSet(SSL_ENABLE_TLS, PR_FALSE) will disable *ALL* versions of TLS,
00281 ** including TLS 1.0 and later.
00282 **
00283 ** The above two properties provide compatibility for applications that use
00284 ** SSL_OptionSet to implement the insecure fallback from TLS 1.x to SSL 3.0.
00285 **
00286 ** SSL_OptionSet(SSL_ENABLE_TLS, PR_TRUE) will enable TLS 1.0, and may also
00287 ** enable some later versions of TLS, if it is necessary to do so in order to
00288 ** keep the set of enabled versions contiguous. For example, if TLS 1.2 is
00289 ** enabled, then after SSL_OptionSet(SSL_ENABLE_TLS, PR_TRUE), TLS 1.0,
00290 ** TLS 1.1, and TLS 1.2 will be enabled, and the call will have no effect on
00291 ** whether SSL 3.0 is enabled. If no later versions of TLS are enabled at the
00292 ** time SSL_OptionSet(SSL_ENABLE_TLS, PR_TRUE) is called, then no later
00293 ** versions of TLS will be enabled by the call.
00294 **
00295 ** SSL_OptionSet(SSL_ENABLE_SSL3, PR_FALSE) will disable SSL 3.0, and will not
00296 ** change the set of TLS versions that are enabled.
00297 **
00298 ** SSL_OptionSet(SSL_ENABLE_SSL3, PR_TRUE) will enable SSL 3.0, and may also
00299 ** enable some versions of TLS if TLS 1.1 or later is enabled at the time of
00300 ** the call, the same way SSL_OptionSet(SSL_ENABLE_TLS, PR_TRUE) works, in
00301 ** order to keep the set of enabled versions contiguous.
00302 */
00303

00436 /*
00437 ** Authenticate certificate hook. Called when a certificate comes in
00438 ** (because of SSL_REQUIRE_CERTIFICATE in SSL_Enable) to authenticate the
00439 ** certificate.
00440 **
00441 ** The authenticate certificate hook must return SECSuccess to indicate the
00442 ** certificate is valid, SECFailure to indicate the certificate is invalid,
00443 ** or SECWouldBlock if the application will authenticate the certificate
00444 ** asynchronously. SECWouldBlock is only supported for non-blocking sockets.
00445 **
00446 ** If the authenticate certificate hook returns SECFailure, then the bad cert
00447 ** hook will be called. The bad cert handler is NEVER called if the
00448 ** authenticate certificate hook returns SECWouldBlock. If the application
00449 ** needs to handle and/or override a bad cert, it should do so before it
00450 ** calls SSL_AuthCertificateComplete (modifying the error it passes to
00451 ** SSL_AuthCertificateComplete as needed).
00452 **
00453 ** See the documentation for SSL_AuthCertificateComplete for more information
00454 ** about the asynchronous behavior that occurs when the authenticate
00455 ** certificate hook returns SECWouldBlock.
00456 **
00457 ** RFC 6066 says that clients should send the bad_certificate_status_response
00458 ** alert when they encounter an error processing the stapled OCSP response.
00459 ** libssl does not provide a way for the authenticate certificate hook to
00460 ** indicate that an OCSP error (SEC_ERROR_OCSP_*) that it returns is an error
00461 ** in the stapled OCSP response or an error in some other OCSP response.
00462 ** Further, NSS does not provide a convenient way to control or determine
00463 ** which OCSP response(s) were used to validate a certificate chain.
00464 ** Consequently, the current version of libssl does not ever send the
00465 ** bad_certificate_status_response alert. This may change in future releases.
00466 */

00966 /* Restart an SSL connection that was paused to do asynchronous certificate
00967  * chain validation (when the auth certificate hook or bad cert handler
00968  * returned SECWouldBlock).
00969  *
00970  * This function only works for non-blocking sockets; Do not use it for
00971  * blocking sockets. Currently, this function works only for the client role of
00972  * a connection; it does not work for the server role.
00973  *
00974  * The application must call SSL_AuthCertificateComplete with 0 as the value of
00975  * the error parameter after it has successfully validated the peer's
00976  * certificate, in order to continue the SSL handshake.
00977  *
00978  * The application may call SSL_AuthCertificateComplete with a non-zero value
00979  * for error (e.g. SEC_ERROR_REVOKED_CERTIFICATE) when certificate validation
00980  * fails, before it closes the connection. If the application does so, an
00981  * alert corresponding to the error (e.g. certificate_revoked) will be sent to
00982  * the peer. See the source code of the internal function
00983  * ssl3_SendAlertForCertError for the current mapping of error to alert. This
00984  * mapping may change in future versions of libssl.
00985  *
00986  * This function will not complete the entire handshake. The application must
00987  * call SSL_ForceHandshake, PR_Recv, PR_Send, etc. after calling this function
00988  * to force the handshake to complete.
00989  *
00990  * On the first handshake of a connection, libssl will wait for the peer's
00991  * certificate to be authenticated before calling the handshake callback,
00992  * sending a client certificate, sending any application data, or returning
00993  * any application data to the application. On subsequent (renegotiation)
00994  * handshakes, libssl will block the handshake unconditionally while the
00995  * certificate is being validated.
00996  *
00997  * libssl may send and receive handshake messages while waiting for the
00998  * application to call SSL_AuthCertificateComplete, and it may call other
00999  * callbacks (e.g, the client auth data hook) before
01000  * SSL_AuthCertificateComplete has been called.
01001  *
01002  * An application that uses this asynchronous mechanism will usually have lower
01003  * handshake latency if it has to do public key operations on the certificate
01004  * chain and/or CRL/OCSP/cert fetching during the authentication, especially if
01005  * it does so in parallel on another thread. However, if the application can
01006  * authenticate the peer's certificate quickly then it may be more efficient
01007  * to use the synchronous mechanism (i.e. returning SECFailure/SECSuccess
01008  * instead of SECWouldBlock from the authenticate certificate hook).
01009  *
01010  * Be careful about converting an application from synchronous cert validation
01011  * to asynchronous certificate validation. A naive conversion is likely to
01012  * result in deadlocks; e.g. the application will wait in PR_Poll for network
01013  * I/O on the connection while all network I/O on the connection is blocked
