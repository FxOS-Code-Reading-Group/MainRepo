00008 /*
00009  * This header contains #defines which tweak the names of various memory
00010  * allocation functions.
00011  *
00012  * There are several types of functions related to memory allocation
00013  * that are meant to be used publicly by the Gecko codebase:
00014  *
00015  * - malloc implementation functions:
00016  *   - malloc
00017  *   - posix_memalign
00018  *   - aligned_alloc
00019  *   - calloc
00020  *   - realloc
00021  *   - free
00022  *   - memalign
00023  *   - valloc
00024  *   - malloc_usable_size
00025  *   - malloc_good_size
00026  *   Some of these functions are specific to some systems, but for
00027  *   convenience, they are treated as being cross-platform, and available
00028  *   as such.
00029  *
00030  * - duplication functions:
00031  *   - strndup
00032  *   - strdup
00033  *   - wcsdup (Windows only)
00034  *
00035  * - jemalloc specific functions:
00036  *   - jemalloc_stats
00037  *   - jemalloc_purge_freed_pages
00038  *   - jemalloc_free_dirty_pages
00039  *   (these functions are native to mozjemalloc, and have compatibility
00040  *   implementations for jemalloc3)
00041  *
00042  * These functions are all exported as part of libmozglue (see
00043  * $(topsrcdir)/mozglue/build/Makefile.in), with a few implementation
00044  * peculiarities:
00045  *
00046  * - On Windows, the malloc implementation functions are all prefixed with
00047  *   "je_", the duplication functions are prefixed with "wrap_", and jemalloc
00048  *   specific functions are left unprefixed. All these functions are however
00049  *   aliased when exporting them, such that the resulting mozglue.dll exports
00050  *   them unprefixed (see $(topsrcdir)/mozglue/build/mozglue.def.in). The
00051  *   prefixed malloc implementation and duplication functions are not
00052  *   exported.
00053  *
00054  * - On MacOSX, the system libc has a zone allocator, which allows us to
00055  *   hook custom malloc implementation functions without exporting them.
00056  *   The malloc implementation functions are all prefixed with "je_" and used
00057  *   this way from the custom zone allocator. They are not exported.
00058  *   Duplication functions are not included, since they will call the custom
00059  *   zone allocator anyways. Jemalloc-specific functions are also left
00060  *   unprefixed.
00061  *
00062  * - On Android, both malloc implementation and duplication functions are
00063  *   prefixed with "__wrap_". Additionally, C++ allocation functions
00064  *   (operator new/delete) are also exported and prefixed with "__wrap_".
00065  *   Jemalloc specific functions are left unprefixed.
00066  *
00067  * - On Gonk, all functions are left unprefixed. Additionally, C++ allocation
00068  *   functions (operator new/delete) are also exported and unprefixed.
00069  *
00070  * - On other systems (mostly Linux), all functions are left unprefixed.
00071  *
00072  * Only Android and Gonk add C++ allocation functions.
00073  *
00074  * Proper exporting of the various functions is done with the MOZ_MEMORY_API
00075  * and MOZ_JEMALLOC_API macros. MOZ_MEMORY_API is meant to be used for malloc
00076  * implementation and duplication functions, while MOZ_JEMALLOC_API is
00077  * dedicated to jemalloc specific functions.
00078  *
00079  *
00080  * All these functions are meant to be called with no prefix from Gecko code.
00081  * In most cases, this is because that's how they are available at runtime.
00082  * However, on Android, "__wrap_" prefixing is left to the build-time linker
00083  * (with -Wl,--wrap), or to the mozmemory.h header for malloc_good_size and
00084  * jemalloc specific functions.
00085  *
00086  *
00087  * Within libmozglue (when MOZ_MEMORY_IMPL is defined), all the functions
00088  * should be suffixed with "_impl" both for declarations and use.
00089  * That is, the implementation declaration for e.g. strdup would look like:
00090  *   char* strdup_impl(const char *)
00091  * That implementation would call malloc by using "malloc_impl".
00092  *
00093  * While mozjemalloc uses these "_impl" suffixed helpers, jemalloc3, being
00094  * third-party code, doesn't, but instead has an elaborate way to mangle
00095  * individual functions. See under "Run jemalloc configure script" in
00096  * $(topsrcdir)/configure.in.
00097  *
00098  *
00099  * When building with replace-malloc support, the above still holds, but
00100  * the malloc implementation and jemalloc specific functions are the
00101  * replace-malloc functions from replace_malloc.c.
00102  *
00103  * The actual jemalloc/mozjemalloc implementation is prefixed with "je_".
00104  *
00105  * Thus, when MOZ_REPLACE_MALLOC is defined, the "_impl" suffixed macros
00106  * expand to "je_" prefixed function when building mozjemalloc or
00107  * jemalloc3/mozjemalloc_compat, where MOZ_JEMALLOC_IMPL is defined.
00108  *
00109  * In other cases, the "_impl" suffixed macros follow the original scheme,
00110  * except on Windows and MacOSX, where they would expand to "je_" prefixed
00111  * functions. Instead, they are left unmodified (malloc_impl expands to
00112  * malloc_impl).
00113  */
