00001 ## Front-End Notes ##
00002 
00003 The email front end code dynamically loads each card implementation, and uses
00004 HTML cached in a document cookie as part of startup. The cached HTML is just
00005 for the first card the user will see when launching the app. By using the
00006 cached HTML, it allows for a quick display to the user while the rest of the
00007 UI and back-end (which runs in a web worker) starts up.
00008 
00009 Once real data is available from the back-end, the cached HTML is removed and
00010 the card with the real data is inserted.
00011 
00012 This means the index.html file is fairly sparse. It just contains one script
00013 tag to inject the cached HTML into the DOM, and that script triggers the load
00014 of the main app JS after injecting the cached HTML.
00015 
00016 The main app JS is built using the Makefile in the email directory. The email
00017 app uses a module loader plugins to load templates: the 'tmpl!...' dependencies,
00018 which in turn depend on 'text!...' dependencies. These loader plugins
00019 participate in the build by leveraging special "pluginBuilder" modules,
00020 tmpl_builder.js and text_builder.js respectively. Since those builder modules
00021 support a few JS build environments, they contain extra code in them, but
00022 they are not actually loaded during the runtime of application, just during the
00023 build process.
00024 
00025 Since it has its own custom build processes, the usual Gaia build processes
00026 do not result in much extra optimization. The Gaia build process for the
00027 localizations still occurs, and the full collection of JS strings for the
00028 translations are inlined, but the HTML DOM is not pre-localized, as each card
00029 HTML is stored in its own .html file that is loaded on demand when the card
00030 is needed.

00032 ## Back-End Code Location ##
00033 
00034 The e-mail back-end is not developed in this repository.  All of the code in
00035 GAIA/apps/email/js/ext/* and its sub-directories comes from:
00036 https://github.com/mozilla-b2g/gaia-email-libs-and-more
00037 
00038 If you want to make fixes to the code, then please issue pull requests against
00039 that repository. This is particularly important since the e-mail back-end uses
00040 staged loading of some roll-up files to get the best performance.
00041 
00042 The e-mail back-end repo copies over a few files to this repository via that
00043 repository's `make install-into-gaia` command. The files are as follows:
00044 
00045 * **js/ext/alameda.js**: used for dynamic loading and module binding for back-end.
00046 * **js/ext/end.js**: Kickstarts back-end loading. If it detects no accounts have
00047 been set up, then it gives the UI a fake data object, then dynamically loads
00048 the bulk of the back-end.
00049 * **js/ext/mailapi/same-frame-setup.js**: the email back-end. It knows how to talk
00050 to the local databases and then dynamically loads account type rollup files.
00051 * **js/ext/mailapi/activesync/configurator.js**: Code for the activesync account
00052 type.
00053 * **js/ext/mailapi/composite/configurator.js**: Code for the "imap+smtp" account
00054 type.
00055 * **js/ext/mailapi/fake/configurator.js**: Code for the "fake" account type, used
00056 for testing.

00059 ## Running in Firefox ##
00060 
00061 Would you like to develop the e-mail app in Firefox instead of b2g desktop?
00062 Yes, you would.
00063 
00064 ### Step 1: Turn on mozTCPSocket ###
00065 
00066 In order to do IMAP stuff, you need to flip the big switch that enables the use
00067 of TCP sockets.  This does not let all webpages open TCP sockets; they still
00068 need the permission we authorize in step 2.
00069 
00070 - Open "about:config" in the browser.
00071 - Right-click anywhere in the config setting listing, and choose "New",
00072   "Boolean" from the context menu.
00073 - Enter "dom.mozTCPSocket.enabled" in the "Enter the preference name" box and
00074   click the "OK" button.
00075 - Select "true" on the next dialog box and click the "OK" button.
00076 
00077 ### Step 2: Authorize mozAnon/mozSystem XHR's and mozTCPSocket use ###
00078 
00079 Now we need to authorize the host you will be running the e-mail app from to be
00080 able to establish IMAP and ActiveSync connections.  IMAP connections depend on
00081 the "tcp-socket" permission to establish TCP connections.  ActiveSync depends on
00082 "systemXHR" in order to issue cross-origin requests and avoid reusing
00083 credentials from other connections.
00084 
00085 Copy/modify the following code to use whatever URL you are hosting the e-mail
00086 app on.  If you are running b2g-desktop in debug mode by doing "make profile
00087 DEBUG=1" then it will host on port 8080.  If you did not explicitly specify a
00088 domain with GAIA_DOMAIN=domain to the make invocation, then your domain is
00089 gaiamobile.org and the app will be found at "email.gaiamobile.org:8080".  Make
00090 sure your /etc/hosts file is pointing this domain at 127.0.0.1!
00091 
00092 Bring up the error console in firefox by hitting control-shift-J.  Paste the
00093 code into the "Code" box and click the "evaluate" button.
00094 
00095 ```
00096 host = 'http://email.gaiamobile.org:8080';
00097 perm = Components.classes["@mozilla.org/permissionmanager;1"]
00098                  .createInstance(Components.interfaces.nsIPermissionManager);
00099 ios = Components.classes["@mozilla.org/network/io-service;1"]
00100                 .getService(Components.interfaces.nsIIOService);
00101 uri = ios.newURI(host, null, null);
00102 perm.add(uri, 'systemXHR', 1);
00103 perm.add(uri, 'tcp-socket', 1);
00104 'Successfully added systemXHR and tcp-socket permissions for '+host;
00105 ```
00106 
00107 At the bottom of the Error Console list, you should see an entry that confirms
00108 the permissions were added.
00109 
00110 ### Step 3: Run Gaia E-Mail! ###
00111 
00112 Start the b2g-desktop instance to serve the e-mail app on port 8080.
00113 
00114 Browse to http://email.gaiamobile.org:8080/ or whatever URL you used above.

00116 ## Build Notes ##
00117 
00118 The email app takes advantage of some features in the gaia build system to do
00119 some optimizations:
00120 
00121 * Apps can define an app-specific Makefile in their directory. This is run
00122 before the rest of the general Gaia build steps. Email uses this to create
00123 a directory in `gaia/build_stage/email` and runs some optimizations around
00124 JS and CSS concatenation.
00125 * The Gaia build system knows to use `gaia/build_stage/email` to do the rest of
00126 the Gaia build steps because email specifies the "dir" in the `gaia_build.json`
00127 in this directory.
00128 * Since the shared resources referenced by email are not listed in the HTML,
00129 but as CSS @imports or via JS module dependencies, the email Makefile
00130 runs `email/build/make_gaia_shared.js` to generate a `gaia_shared.json` file
00131 in the `gaia/build_stage/email` directory to list out the shared items use by
00132 the email app. `gaia_shared.json` is used by the general Gaia build system to
00133 know what shared resources to keep. The Gaia build system also does HTML file
00134 scanning to find shared resources too.
00135 
00136 For DEBUG=1 builds, the email source directory is used as-is, and the shared
00137 resources are magically linked in via the Gaia build system.
00138 
00139 If you want to give snapshots of builds, say for UX reviews, you should use
00140 the contents of the `gaia/build_stage/email` directory as it will be a fully
00141 functional snapshot.
