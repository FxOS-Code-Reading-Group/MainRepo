00174  *
00175  * There is no rotate operation in the C Language so the construct (a << 4) |
00176  * (a >> 28) is used instead. Most compilers convert this to a rotate
00177  * instruction but some versions of MSVC don't without a little help.  To get

00194 /*
00195  * Low-level memory management in SpiderMonkey:
00196  *
00197  *  ** Do not use the standard malloc/free/realloc: SpiderMonkey allows these
00198  *     to be redefined (via JS_USE_CUSTOM_ALLOCATOR) and Gecko even #define's
00199  *     these symbols.
00200  *
00201  *  ** Do not use the builtin C++ operator new and delete: these throw on
00202  *     error and we cannot override them not to.
00203  *
00204  * Allocation:
00205  *
00206  * - If the lifetime of the allocation is tied to the lifetime of a GC-thing
00207  *   (that is, finalizing the GC-thing will free the allocation), call one of
00208  *   the following functions:
00209  *
00210  *     JSContext::{malloc_,realloc_,calloc_,new_}
00211  *     JSRuntime::{malloc_,realloc_,calloc_,new_}
00212  *
00213  *   These functions accumulate the number of bytes allocated which is used as
00214  *   part of the GC-triggering heuristic.
00215  *
00216  *   The difference between the JSContext and JSRuntime versions is that the
00217  *   cx version reports an out-of-memory error on OOM. (This follows from the
00218  *   general SpiderMonkey idiom that a JSContext-taking function reports its
00219  *   own errors.)
00220  *
00221  * - Otherwise, use js_malloc/js_realloc/js_calloc/js_free/js_new
00222  *
00223  * Deallocation:
00224  *
00225  * - Ordinarily, use js_free/js_delete.
00226  *
00227  * - For deallocations during GC finalization, use one of the following
00228  *   operations on the FreeOp provided to the finalizer:
00229  *
00230  *     FreeOp::{free_,delete_}
00231  *
00232  *   The advantage of these operations is that the memory is batched and freed
00233  *   on another thread.
00234  */

00405 /*
00406  * Given a raw hash code, h, return a number that can be used to select a hash
00407  * bucket.
00408  *
00409  * This function aims to produce as uniform an output distribution as possible,
00410  * especially in the most significant (leftmost) bits, even though the input
00411  * distribution may be highly nonrandom, given the constraints that this must
00412  * be deterministic and quick to compute.
00413  *
00414  * Since the leftmost bits of the result are best, the hash bucket index is
00415  * computed by doing ScrambleHashCode(h) / (2^32/N) or the equivalent
00416  * right-shift, not ScrambleHashCode(h) % N or the equivalent bit-mask.
00417  *
00418  * FIXME: OrderedHashTable uses a bit-mask; see bug 775896.
00419  */
00420 inline HashNumber
00421 ScrambleHashCode(HashNumber h)
00422 {
00423     /*
00424      * Simply returning h would not cause any hash tables to produce wrong
00425      * answers. But it can produce pathologically bad performance: The caller
00426      * right-shifts the result, keeping only the highest bits. The high bits of
00427      * hash codes are very often completely entropy-free. (So are the lowest
00428      * bits.)
00429      *
00430      * So we use Fibonacci hashing, as described in Knuth, The Art of Computer
00431      * Programming, 6.4. This mixes all the bits of the input hash code h.
00432      * 
00433      * The value of goldenRatio is taken from the hex
00434      * expansion of the golden ratio, which starts 1.9E3779B9....
00435      * This value is especially good if values with consecutive hash codes
00436      * are stored in a hash table; see Knuth for details.
00437      */
00438     static const HashNumber goldenRatio = 0x9E3779B9U;
00439     return h * goldenRatio;
00440 }
