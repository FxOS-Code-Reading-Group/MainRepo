00035 /*
00036  * MediaStreamGraph is a framework for synchronized audio/video processing
00037  * and playback. It is designed to be used by other browser components such as
00038  * HTML media elements, media capture APIs, real-time media streaming APIs,
00039  * multitrack media APIs, and advanced audio APIs.
00040  *
00041  * The MediaStreamGraph uses a dedicated thread to process media --- the media
00042  * graph thread. This ensures that we can process media through the graph
00043  * without blocking on main-thread activity. The media graph is only modified
00044  * on the media graph thread, to ensure graph changes can be processed without
00045  * interfering with media processing. All interaction with the media graph
00046  * thread is done with message passing.
00047  *
00048  * APIs that modify the graph or its properties are described as "control APIs".
00049  * These APIs are asynchronous; they queue graph changes internally and
00050  * those changes are processed all-at-once by the MediaStreamGraph. The
00051  * MediaStreamGraph monitors the main thread event loop via nsIAppShell::RunInStableState
00052  * to ensure that graph changes from a single event loop task are always
00053  * processed all together. Control APIs should only be used on the main thread,
00054  * currently; we may be able to relax that later.
00055  *
00056  * To allow precise synchronization of times in the control API, the
00057  * MediaStreamGraph maintains a "media timeline". Control APIs that take or
00058  * return times use that timeline. Those times never advance during
00059  * an event loop task. This time is returned by MediaStreamGraph::GetCurrentTime().
00060  *
00061  * Media decoding, audio processing and media playback use thread-safe APIs to
00062  * the media graph to ensure they can continue while the main thread is blocked.
00063  *
00064  * When the graph is changed, we may need to throw out buffered data and
00065  * reprocess it. This is triggered automatically by the MediaStreamGraph.
00066  */
