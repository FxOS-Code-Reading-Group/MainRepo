00009 #ifndef mozilla_Move_h
00010 #define mozilla_Move_h
00011 
00012 namespace mozilla {

00014 /*
00015  * "Move" References
00016  *
00017  * Some types can be copied much more efficiently if we know the original's
00018  * value need not be preserved --- that is, if we are doing a "move", not a
00019  * "copy". For example, if we have:
00020  *
00021  *   Vector<T> u;
00022  *   Vector<T> v(u);
00023  *
00024  * the constructor for v must apply a copy constructor to each element of u ---
00025  * taking time linear in the length of u. However, if we know we will not need u
00026  * any more once v has been initialized, then we could initialize v very
00027  * efficiently simply by stealing u's dynamically allocated buffer and giving it
00028  * to v --- a constant-time operation, regardless of the size of u.
00029  *
00030  * Moves often appear in container implementations. For example, when we append
00031  * to a vector, we may need to resize its buffer. This entails moving each of
00032  * its extant elements from the old, smaller buffer to the new, larger buffer.
00033  * But once the elements have been migrated, we're just going to throw away the
00034  * old buffer; we don't care if they still have their values. So if the vector's
00035  * element type can implement "move" more efficiently than "copy", the vector
00036  * resizing should by all means use a "move" operation. Hash tables also need to
00037  * be resized.
00038  *
00039  * The details of the optimization, and whether it's worth applying, vary from
00040  * one type to the next. And while some constructor calls are moves, many really
00041  * are copies, and can't be optimized this way. So we need:
00042  *
00043  * 1) a way for a particular invocation of a copy constructor to say that it's
00044  *    really a move, and that the value of the original isn't important
00045  *    afterwards (although it must still be safe to destroy); and
00046  *
00047  * 2) a way for a type (like Vector) to announce that it can be moved more
00048  *    efficiently than it can be copied, and provide an implementation of that
00049  *    move operation.
00050  *
00051  * The Move(T&) function takes a reference to a T, and returns a MoveRef<T>
00052  * referring to the same value; that's 1). A MoveRef<T> is simply a reference
00053  * to a T, annotated to say that a copy constructor applied to it may move that
00054  * T, instead of copying it. Finally, a constructor that accepts an MoveRef<T>
00055  * should perform a more efficient move, instead of a copy, providing 2).
00056  *
00057  * So, where we might define a copy constructor for a class C like this:
00058  *
00059  *   C(const C& rhs) { ... copy rhs to this ... }
00060  *
00061  * we would declare a move constructor like this:
00062  *
00063  *   C(MoveRef<C> rhs) { ... move rhs to this ... }
00064  *
00065  * And where we might perform a copy like this:
00066  *
00067  *   C c2(c1);
00068  *
00069  * we would perform a move like this:
00070  *
00071  *   C c2(Move(c1))
00072  *
00073  * Note that MoveRef<T> implicitly converts to T&, so you can pass a MoveRef<T>
00074  * to an ordinary copy constructor for a type that doesn't support a special
00075  * move constructor, and you'll just get a copy.  This means that templates can
00076  * use Move whenever they know they won't use the original value any more, even
00077  * if they're not sure whether the type at hand has a specialized move
00078  * constructor.  If it doesn't, the MoveRef<T> will just convert to a T&, and
00079  * the ordinary copy constructor will apply.
00080  *
00081  * A class with a move constructor can also provide a move assignment operator,
00082  * which runs this's destructor, and then applies the move constructor to
00083  * *this's memory. A typical definition:
00084  *
00085  *   C& operator=(MoveRef<C> rhs) {
00086  *     this->~C();
00087  *     new(this) C(rhs);
00088  *     return *this;
00089  *   }
00090  *
00091  * With that in place, one can write move assignments like this:
00092  *
00093  *   c2 = Move(c1);
00094  *
00095  * This destroys c1, moves c1's value to c2, and leaves c1 in an undefined but
00096  * destructible state.
00097  *
00098  * This header file defines MoveRef and Move in the mozilla namespace.  It's up
00099  * to individual containers to annotate moves as such, by calling Move; and it's
00100  * up to individual types to define move constructors.
00101  *
00102  * One hint: if you're writing a move constructor where the type has members
00103  * that should be moved themselves, it's much nicer to write this:
00104  *
00105  *   C(MoveRef<C> c) : x(Move(c->x)), y(Move(c->y)) { }
00106  *
00107  * than the equivalent:
00108  *
00109  *   C(MoveRef<C> c) { new(&x) X(Move(c->x)); new(&y) Y(Move(c->y)); }
00110  *
00111  * especially since GNU C++ fails to notice that this does indeed initialize x
00112  * and y, which may matter if they're const.
00113  */
00114 template<typename T>
00115 class MoveRef
00116 {
00117     T* pointer;
00118 
00119   public:
00120     explicit MoveRef(T& t) : pointer(&t) { }
00121     T& operator*() const { return *pointer; }
00122     T* operator->() const { return pointer; }
00123     operator T& () const { return *pointer; }
00124 };
00125 
00126 template<typename T>
00127 inline MoveRef<T>
00128 Move(T& t)
00129 {
00130   return MoveRef<T>(t);
00131 }
00132 
00133 template<typename T>
00134 inline MoveRef<T>
00135 Move(const T& t)
00136 {
00137   // With some versions of gcc, for a class C, there's an (incorrect) ambiguity
00138   // between the C(const C&) constructor and the default C(C&&) C++11 move
00139   // constructor, when the constructor is called with a const C& argument.
00140   //
00141   // This ambiguity manifests with the Move implementation above when Move is
00142   // passed const U& for some class U.  Calling Move(const U&) returns a
00143   // MoveRef<const U&>, which is then commonly passed to the U constructor,
00144   // triggering an implicit conversion to const U&.  gcc doesn't know whether to
00145   // call U(const U&) or U(U&&), so it wrongly reports a compile error.
00146   //
00147   // http://gcc.gnu.org/bugzilla/show_bug.cgi?id=50442 has since been fixed, so
00148   // this is no longer an issue for up-to-date compilers.  But there's no harm
00149   // in keeping it around for older compilers, so we might as well.  See also
00150   // bug 686280.
00151   return MoveRef<T>(const_cast<T&>(t));
00152 }

00157 Swap(T& t, T& u)
00158 {
00159   T tmp(Move(t));

00161   u = Move(tmp);
