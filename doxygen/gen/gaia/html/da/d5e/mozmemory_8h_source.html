00054 /*
00055  * On some operating systems (Mac), we use madvise(MADV_FREE) to hand pages
00056  * back to the operating system.  On Mac, the operating system doesn't take
00057  * this memory back immediately; instead, the OS takes it back only when the
00058  * machine is running out of physical memory.
00059  *
00060  * This is great from the standpoint of efficiency, but it makes measuring our
00061  * actual RSS difficult, because pages which we've MADV_FREE'd shouldn't count
00062  * against our RSS.
00063  *
00064  * This function explicitly purges any MADV_FREE'd pages from physical memory,
00065  * causing our reported RSS match the amount of memory we're actually using.
00066  *
00067  * Note that this call is expensive in two ways.  First, it may be slow to
00068  * execute, because it may make a number of slow syscalls to free memory.  This
00069  * function holds the big jemalloc locks, so basically all threads are blocked
00070  * while this function runs.
00071  *
00072  * This function is also expensive in that the next time we go to access a page
00073  * which we've just explicitly decommitted, the operating system has to attach
00074  * to it a physical page!  If we hadn't run this function, the OS would have
00075  * less work to do.
00076  *
00077  * If MALLOC_DOUBLE_PURGE is not defined, this function does nothing.
00078  */
00079 MOZ_JEMALLOC_API void jemalloc_purge_freed_pages();
00080 
00081 /*
00082  * Free all unused dirty pages in all arenas. Calling this function will slow
00083  * down subsequent allocations so it is recommended to use it only when
00084  * memory needs to be reclaimed at all costs (see bug 805855). This function
00085  * provides functionality similar to mallctl("arenas.purge") in jemalloc 3.
00086  */
