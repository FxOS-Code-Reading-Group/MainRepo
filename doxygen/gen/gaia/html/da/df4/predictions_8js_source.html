00003 //
00004 // This is a JavaScript predictive text engine: given a dictionary, a data
00005 // structure that specifies which keys are near which other keys, and
00006 // a string of user input, it guesses what the user meant to type or, (if
00007 // if cannot find a word loosely matching the input) what the user is planning
00008 // to type.
00009 //
00010 // This module defines a single global variable Predictions which is an
00011 // object with the following methods:
00012 //
00013 //   setDictionary: specifies the dictionary to use
00014 //
00015 //   setNearbyKeys: specifies which keys are near which other keys
00016 //
00017 //   predict: given an input string, asynchronously find corrections or
00018 //      predictions and pass them to a specified callback.
00019 //
00020 // The word prediction / auto-correction algorithm works by finding loose
00021 // matches for the user's input. The hard part of this process is getting
00022 // the "loose" part right. For each character in the user's input, we'll
00023 // match variant forms of the character to handle case differences and
00024 // diacritics. In languages that use apostrophes and hyphens, we'll match
00025 // those at any point, even if the user omits them. Because we've been
00026 // passed a data structure that tells us which keys are near each other (and
00027 // how near they are) we use that information to correct typing errors.
00028 // Other loose matching techniques we try include transpositions, insertions
00029 // and deletions.
00030 //
00031 // During this loose matching process, we associate a weight with each
00032 // candidate we're considering. This weight is based on the word frequency
00033 // information from the dictionary file, but it is reduced when we make
00034 // corrections to the user's input, to reflect our decreased confidence in
00035 // the suggestion. Substituting variant forms and inserting apostrophes does
00036 // not reduce the weight by much at all. Substituting a nearby key reduces
00037 // the weight moderately (depending on how near the keys are), and
00038 // insertions and deletions and transpositions reduce the weight
00039 // significantly.
00040 //
00041 // The core part of the matching algorithm is in the process() function
00042 // nested inside the predict() method. To fully understand it, however, you
00043 // also need to understand the weighted Ternary Search Tree data structure
00044 // used to represent the dictionary.
00045 //
00046 // Each node of the tree represents a single character, and is part of
00047 // an ordinary binary tree. Nodes representing characters that are less
00048 // than the current node are found by following the left pointer, and nodes
00049 // that repesent characters that are greater than the character in the
00050 // current node are found by following the right pointer.
00051 //
00052 // Our data structure is called a ternary tree because in addition to these
00053 // left and right pointers, each node also has a center pointer that points
00054 // to the subtree of nodes that represent the next character of the
00055 // word. You can think of all the nodes found by recursively traversing
00056 // left and right from the current node as being on the same level, and the
00057 // node found by by traversing to the center pointer as being on the next
00058 // level.
00059 //
00060 // Ternary trees are good for representing dictionaries and checking
00061 // whether a string appears in the dictionary. In order to make
00062 // predictions, however, our dictionary also contains word frequency
00063 // information, and we use it to predict more frequently used words instead
00064 // of less frequently used words. So instead of using a generic ternary
00065 // search tree, we use a weighted ternary search tree. In addition to the
00066 // left, center, and right pointers, each node also has a next pointer that
00067 // points to the node that represents the next most likely letter at the
00068 // current position. In effect, all of the nodes that are on the same level
00069 // form a linked list sorted from most likely to least likely.
00070 //
00071 // To make this linked list work, the tree uses non-traditional
00072 // balancing. Instead of balancing the binary tree so that there are
00073 // approximately equal numbers of nodes on the left and the right, the tree
00074 // is balanced so that when you follow the center pointer from one level to
00075 // the next level, the node represents the most likely letter on that
00076 // level. Each center pointer points to the head of a linked list.
00077 //
00078 // In addition to holding a character, each node of the tree also stores a
00079 // frequency value. For each node, this number is the frequency of the most
00080 // frequent word beneath that node. If you recursively follow the center
00081 // pointer of a node, you'll find that most frequent word.
00082 //
00083 // It turns out that our matching algorithm does not actually use the left
00084 // and right pointers of the tree at all, so they are omitted from the
00085 // dictionary to dramatically reduce its file size. At any given "level" of
00086 // the tree, we find the possible letters by following the next pointer. And
00087 // at any given node, we move to the next level by following the center
00088 // pointer.
00089 //
00090 // You can read more about the data structure in the script that generates the
00091 // dictionary files:
00092 //
00093 //   gaia/dictionaries/xml2dict.py.
00094 //
00095 // Also see the following online resources which include helpful diagrams:
00096 //
00097 //   http://en.wikipedia.org/wiki/Ternary_search_tree
00098 //   http://www.strchr.com/ternary_dags
00099 //   http://www.strchr.com/dawg_predictive
00100 //
00101 // Note that this implementation does not convert the TST into a DAG to
00102 // share common suffix nodes as described in the strchr.com blog
00103 // posts. (That is an optimization that may be possible later: the shared
00104 // suffix nodes can't hold correct word frequencies, so the search
00105 // algorithm would have to be modified to carry the correct frequency
00106 // through while processing a shared suffix.)

00433           if (characterTable.hasOwnProperty(c))
00434             continue outer;
00435         }

00444     // Add a candidate to the list of promising candidates if frequency *
00445     // multiplier is high enough. A candidate is a pointer (byte offset) to
00446     // a node in the tree, the portion of the user's input that has not yet
00447     // been considered, the output string that has been generated so far, a
00448     // number based on the highest frequency word that begins with the
00449     // output we've generated, a multipler that adjusts that frequency based
00450     // on how much we've modified the user's input, and a number that
00451     // indicates how many times we've already corrected the user's input for
00452     // this candidate.
00453     function addCandidate(pointer, remaining, output,
00454                           multiplier, frequency, corrections)
00455     {
00456       var weight = frequency * multiplier;
00457 
00458       // If no major corrections have been made to this candidate, then
00459       // artificially increase its weight so that it appears in the
00460       // candidates list before any corrected candidates. This should
00461       // ensure that if the user is typing an infrequent word we don't
00462       // bump the actual word off the list if there are lots of frequent
00463       // words that have similar spellings. The artificial weight does
00464       // not carry through to the list of words, so more frequent words may
00465       // still be predicted instead of the user's input when the user is
00466       // typing an infrequent word. But we shouldn't ever not be able to
00467       // find the user's input as a valid word.  Adding letters to the
00468       // end of partial input does not count as a correction so we also
00469       // test the multiplier so that we don't boost the weight of every
00470       // extension. But we do allow one letter to be added on and still
00471       // get the extra weight.
00472       if (corrections === 0 &&
00473           multiplier > wordExtensionMultiplier * wordExtensionMultiplier) {
00474         weight += 100;
00475       }
00476 
00477       // If this candidate could never become a word, don't add it
00478       if (weight <= words.threshold)
00479         return;
00480 
00481       candidates.add({
00482         pointer: pointer,
00483         input: remaining,
00484         output: output,
00485         multiplier: multiplier,
00486         weight: weight,
00487         corrections: corrections
00488       }, weight);
00489     }

00491     // Add a word to the priority queue of words
00492     function addWord(word, weight) {
00493       // If the input was capitalized, capitalize the word
00494       if (capitalize)
00495         word = word[0].toUpperCase() + word.substring(1);
00496 
00497       // Make sure we don't already have the word in the queue
00498       for (var i = 0, n = words.items.length; i < n; i++) {
00499         if (words.items[i][0] === word) {
00500           // If the version we already have has higher weight, skip this one
00501           if (words.priorities[i] >= weight)
00502             return;
00503           else // otherwise, remove the existing lower-weight copy
00504             words.removeItemAt(i);
00505           break;
00506         }
00507       }
00508 
00509       words.add([word, weight], weight);
00510     }

00513     // process it. (This will often add more candidates to the list). After
00514     // we've processed a batch of candidates this way, use setTimeout() to
00515     // schedule the processing of the next batch after returning to the

00868   // A priority queue with a maximum size.
00869   //
00870   // add() inserts an item at a position according to its priority. It
00871   // returns true if the item was inserted or false if the item's priority
00872   // was too low for a spot in the fixed-size queue.
00873   //
00874   // remove() removes and returns the highest priority item in the queue or
00875   // null if there are no items
00876   //
00877   // threshold is 0 if the queue is not yet full. Otherwise it is the
00878   // priority of the lowest-priority item in the queue. Items with
00879   // priorities higher lower than this will never be added to the queue.
00880   //
00881   // items is the sorted array of items, with the highest priority item
00882   // first.
00883   //
00884   function BoundedPriorityQueue(maxSize) {
00885     this.maxSize = maxSize;
00886     this.threshold = 0;
00887     this.items = [];
00888     this.priorities = [];
00889   }

00891   BoundedPriorityQueue.prototype.add = function add(item, priority) {
00892     // If the array is full we have to reject this item or make room for it
00893     if (this.items.length === this.maxSize) {
00894       if (priority <= this.threshold) { // Reject the item.
00895         return false;
00896       }
00897       else {                            // Make room for it.
00898         this.items.pop();
00899         this.priorities.pop();
00900       }
00901     }
00902 
00903     // Search to find the insertion point for this new item
00904     var index;
00905     if (this.priorities.length > 60) {
00906       // Binary search only for relatively long arrays.
00907       // See http://jsperf.com/linear-or-binary-search for perf data.
00908       var start = 0, end = this.priorities.length;
00909       while (start !== end) {
00910         var mid = Math.floor((start + end) / 2);
00911         if (priority > this.priorities[mid]) {
00912           end = mid;
00913         }
00914         else {
00915           start = mid + 1;
00916         }
00917       }
00918       index = start;
00919     }
00920     else {
00921       // Linear search for small arrays
00922       for (var i = 0, n = this.priorities.length; i < n; i++) {
00923         if (priority > this.priorities[i])
00924           break;
00925       }
00926       index = i;
00927     }
00928 
00929     // Insert the new item at that position
00930     this.items.splice(index, 0, item);
00931     this.priorities.splice(index, 0, priority);
00932 
00933     // Update the threshold
00934     this.threshold = this.priorities[this.maxSize - 1] || 0;
00935   };

00963   // Cache the key/value pair
00964   LRUCache.prototype.add = function add(key, value) {
00965     // If the key is already in the cache, adjust the size since we'll
00966     // be incrementing below
00967     if (key in this.map) {
00968       this.size--;
00969     }
00970 
00971     // Now insert the item
00972     this.map[key] = value;

00976     // for/in. This should be the least recently used because the get()
00977     // method deletes and reinserts
00978     if (this.size > this.maxsize) {
00979       for (var p in this.map) {

00981         break;
