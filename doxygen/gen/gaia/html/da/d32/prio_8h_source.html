00429 /*
00430  **************************************************************************
00431  * Layering file descriptors
00432  *
00433  * File descriptors may be layered. Each layer has it's own identity.
00434  * Identities are allocated by the runtime and are to be associated
00435  * (by the layer implementor) with all layers that are of that type.
00436  * It is then possible to scan the chain of layers and find a layer
00437  * that one recongizes and therefore predict that it will implement
00438  * a desired protocol.
00439  *
00440  * There are three well-known identities:
00441  *      PR_INVALID_IO_LAYER => an invalid layer identity, for error return
00442  *      PR_TOP_IO_LAYER     => the identity of the top of the stack
00443  *      PR_NSPR_IO_LAYER    => the identity used by NSPR proper
00444  * PR_TOP_IO_LAYER may be used as a shorthand for identifying the topmost
00445  * layer of an existing stack. Ie., the following two constructs are
00446  * equivalent.
00447  *
00448  *      rv = PR_PushIOLayer(stack, PR_TOP_IO_LAYER, my_layer);
00449  *      rv = PR_PushIOLayer(stack, PR_GetLayersIdentity(stack), my_layer)
00450  *
00451  * A string may be associated with the creation of the identity. It
00452  * will be copied by the runtime. If queried the runtime will return
00453  * a reference to that copied string (not yet another copy). There
00454  * is no facility for deleting an identity.
00455  **************************************************************************
00456  */

01954 /*
01955 **************************************************************************
01956 **
01957 ** Pollable events
01958 **
01959 ** A pollable event is a special kind of file descriptor.
01960 ** The only I/O operation you can perform on a pollable event
01961 ** is to poll it with the PR_POLL_READ flag.  You can't
01962 ** read from or write to a pollable event.
01963 **
01964 ** The purpose of a pollable event is to combine event waiting
01965 ** with I/O waiting in a single PR_Poll call.  Pollable events
01966 ** are implemented using a pipe or a pair of TCP sockets
01967 ** connected via the loopback address, therefore setting and
01968 ** waiting for pollable events are expensive operating system
01969 ** calls.  Do not use pollable events for general thread
01970 ** synchronization. Use condition variables instead.
01971 **
01972 ** A pollable event has two states: set and unset.  Events
01973 ** are not queued, so there is no notion of an event count.
01974 ** A pollable event is either set or unset.
01975 **
01976 ** A new pollable event is created by a PR_NewPollableEvent
01977 ** call and is initially in the unset state.
01978 **
01979 ** PR_WaitForPollableEvent blocks the calling thread until
01980 ** the pollable event is set, and then it atomically unsets
01981 ** the pollable event before it returns.
01982 **
01983 ** To set a pollable event, call PR_SetPollableEvent.
01984 **
01985 ** One can call PR_Poll with the PR_POLL_READ flag on a pollable
01986 ** event.  When the pollable event is set, PR_Poll returns with
01987 ** the PR_POLL_READ flag set in the out_flags.
01988 **
01989 ** To close a pollable event, call PR_DestroyPollableEvent
01990 ** (not PR_Close).
01991 **
01992 **************************************************************************
01993 */
