00023 /*
00024  * Protecting non-Value, non-JSObject *, non-JSString * values from collection
00025  *
00026  * Most of the time, the garbage collector's conservative stack scanner works
00027  * behind the scenes, finding all live values and protecting them from being
00028  * collected. However, when JSAPI client code obtains a pointer to data the
00029  * scanner does not know about, owned by an object the scanner does know about,
00030  * Care Must Be Taken.
00031  *
00032  * The scanner recognizes only a select set of types: pointers to JSObjects and
00033  * similar things (JSFunctions, and so on), pointers to JSStrings, and Values.
00034  * So while the scanner finds all live |JSString| pointers, it does not notice
00035  * |jschar| pointers.
00036  *
00037  * So suppose we have:
00038  *
00039  *   void f(JSString *str) {
00040  *     const jschar *ch = JS_GetStringCharsZ(str);
00041  *     ... do stuff with ch, but no uses of str ...;
00042  *   }
00043  *
00044  * After the call to |JS_GetStringCharsZ|, there are no further uses of
00045  * |str|, which means that the compiler is within its rights to not store
00046  * it anywhere. But because the stack scanner will not notice |ch|, there
00047  * is no longer any live value in this frame that would keep the string
00048  * alive. If |str| is the last reference to that |JSString|, and the
00049  * collector runs while we are using |ch|, the string's array of |jschar|s
00050  * may be freed out from under us.
00051  *
00052  * Note that there is only an issue when 1) we extract a thing X the scanner
00053  * doesn't recognize from 2) a thing Y the scanner does recognize, and 3) if Y
00054  * gets garbage-collected, then X gets freed. If we have code like this:
00055  *
00056  *   void g(JSObject *obj) {
00057  *     JS::Value x;
00058  *     JS_GetProperty(obj, "x", &x);
00059  *     ... do stuff with x ...
00060  *   }
00061  *
00062  * there's no problem, because the value we've extracted, x, is a Value, a
00063  * type that the conservative scanner recognizes.
00064  *
00065  * Conservative GC frees us from the obligation to explicitly root the types it
00066  * knows about, but when we work with derived values like |ch|, we must root
00067  * their owners, as the derived value alone won't keep them alive.
00068  *
00069  * A JS::Anchor is a kind of GC root that allows us to keep the owners of
00070  * derived values like |ch| alive throughout the Anchor's lifetime. We could
00071  * fix the above code as follows:
00072  *
00073  *   void f(JSString *str) {
00074  *     JS::Anchor<JSString *> a_str(str);
00075  *     const jschar *ch = JS_GetStringCharsZ(str);
00076  *     ... do stuff with ch, but no uses of str ...;
00077  *   }
00078  *
00079  * This simply ensures that |str| will be live until |a_str| goes out of scope.
00080  * As long as we don't retain a pointer to the string's characters for longer
00081  * than that, we have avoided all garbage collection hazards.
00082  */
00083 template<typename T> class AnchorPermitted;
00084 template<> class AnchorPermitted<JSObject *> { };
00085 template<> class AnchorPermitted<const JSObject *> { };
00086 template<> class AnchorPermitted<JSFunction *> { };
00087 template<> class AnchorPermitted<const JSFunction *> { };
00088 template<> class AnchorPermitted<JSString *> { };
00089 template<> class AnchorPermitted<const JSString *> { };
00090 template<> class AnchorPermitted<Value> { };
00091 template<> class AnchorPermitted<const JSScript *> { };
00092 template<> class AnchorPermitted<JSScript *> { };

00097   public:
00098     Anchor() { }
00099     explicit Anchor(T t) { hold = t; }
00100     inline ~Anchor();

00102     const T &get() const { return hold; }
00103     void set(const T &t) { hold = t; }

00113 template<typename T>
00114 inline Anchor<T>::~Anchor()
00115 {
00116 #ifdef __GNUC__
00117     /*
00118      * No code is generated for this. But because this is marked 'volatile', G++ will
00119      * assume it has important side-effects, and won't delete it. (G++ never looks at
00120      * the actual text and notices it's empty.) And because we have passed |hold| to
00121      * it, GCC will keep |hold| alive until this point.
00122      *
00123      * The "memory" clobber operand ensures that G++ will not move prior memory
00124      * accesses after the asm --- it's a barrier. Unfortunately, it also means that
00125      * G++ will assume that all memory has changed after the asm, as it would for a
00126      * call to an unknown function. I don't know of a way to avoid that consequence.
00127      */
00128     asm volatile("":: "g" (hold) : "memory");
00129 #else
00130     /*
00131      * An adequate portable substitute, for non-structure types.
00132      *
00133      * The compiler promises that, by the end of an expression statement, the
00134      * last-stored value to a volatile object is the same as it would be in an
00135      * unoptimized, direct implementation (the "abstract machine" whose behavior the
00136      * language spec describes). However, the compiler is still free to reorder
00137      * non-volatile accesses across this store --- which is what we must prevent. So
00138      * assigning the held value to a volatile variable, as we do here, is not enough.
00139      *
00140      * In our case, however, garbage collection only occurs at function calls, so it
00141      * is sufficient to ensure that the destructor's store isn't moved earlier across
00142      * any function calls that could collect. It is hard to imagine the compiler
00143      * analyzing the program so thoroughly that it could prove that such motion was
00144      * safe. In practice, compilers treat calls to the collector as opaque operations
00145      * --- in particular, as operations which could access volatile variables, across
00146      * which this destructor must not be moved.
00147      *
00148      * ("Objection, your honor!  *Alleged* killer whale!")
00149      *
00150      * The disadvantage of this approach is that it does generate code for the store.
00151      * We do need to use Anchors in some cases where cycles are tight.
00152      *
00153      * Note that there is a Anchor<Value>::~Anchor() specialization in Value.h.
00154      */
00155     volatile T sink;
00156     sink = hold;
00157 #endif  /* defined(__GNUC__) */
00158 }
